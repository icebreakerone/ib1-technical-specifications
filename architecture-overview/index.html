    Trust Framework Architecture Overview

    This overview explains how the IB1 specifications combine into a coherent architecture, and how to choose and apply them inside a Scheme. Schemes choose the specifications they need; only explicit dependencies are mandatory.

    Core definitions (for this overview) - Registry: The authoritative catalog of URIs for roles, licenses, schemas, messages, and scheme metadata; also holds technical profiles and version history. - Directory: The public index of Members and Applications, including roles, assurance levels, endpoints (APIs, message delivery), and Catalog submissions. - Catalog: DCAT-based metadata entries describing datasets and data services; published via the Directory using the Catalog Publishing process. - Member certificate: Client/signing certificate issued by the Directory binding a Member and Application URL (and roles) for mTLS and signing. - Server certificate: TLS certificate authenticating a service endpoint operated by a Member; either Directory-issued or public-CA with Directory allowlist. - License: Machine-readable legal instrument registered in the Registry; reused as OAuth scopes and referenced in catalog, permission, and provenance records. - Role: Registry-defined capability identifier used for access control; embedded in Directory entries and Member certificates. - Message subject: Registry URI that defines a message type and points to its JSON Schema for Message Delivery payloads.

    Design choices that shape the stack

    Stable, URL-based identifiers are the spine of the Trust Framework. Everything meaningful—roles, licenses, message subjects, data schemas, scheme identifiers—lives in the Registry with a permanent URI. Once a Scheme picks an element, that URI is reused everywhere else: in Directory records, certificate extensions, OAuth scopes, catalog entries, provenance steps, and message payloads. This keeps agreements explicit and machine-followable.

    Identity is anchored in certificates that embed Directory URLs and roles. When two systems talk over mTLS, the certificate already tells you “who” and “in what capacity” without a side-band client registry. The same pattern applies to signing provenance: the certificate ties the signature to a specific Application and Member.

    Discovery is a first-class capability. If you need to know who an actor is or where to send a message, you query the Directory. If you need to know what data exists or how to call it, you query the Catalog. Message Delivery and OAuth flows both start with Directory metadata, so integrations do not rely on hard-coded endpoints.

    Payload definitions are transport-agnostic. Licenses, permission records, provenance steps, and message schemas are defined once, then reused whether the exchange happens over REST, Message Delivery, or file transfer. This lets Schemes choose transports based on operational fit while keeping shared semantics.

    All specifications are modular. Adopting one capability brings along consistent identifiers, certificate bindings, and Registry-backed schemas, but does not force unrelated pieces. Dependencies are explicit and minimal so a Scheme can assemble only what it needs.

    Governance and Definitions

    • Registry & Versioning (registry, registry-versioning, registry-process-resources): Holds authoritative URIs for trust frameworks, schemes, technical profiles, roles, licenses, and message/data schemas. Versioning guarantees stability of identifiers referenced across the ecosystem.
    • Machine-readable Licenses (machine-readable-data-licenses): Encodes the legal basis for data access; license URIs are reused as OAuth scopes and inside catalog, permission, and provenance records.
    • Role-based Access Control (role-based-access-control): Defines roles in the Registry and reuses them in certificates, Directory entries, and service enforcement.
    • Assurance and Sensitivity (generic-organizational-assurance-levels, generic-dataset-assurance-levels, generic-sensitivity-classes): Optional policy levers Schemes can reference from the Registry to grade participants and data.

    Participant and Service Identity

    • Directory (directory): Public index of Members and Applications, publishing RDF/JSON metadata and message delivery endpoints. Provides discovery of roles, schemes, assurance levels, and Catalog entries.
    • Certificates for participants (member-identity-digital-certificates): Long-lived client/signing certificates encode Application, Member URL, and roles from the Directory; used for mTLS and signing provenance.
    • Server authentication (directory-issued-server-tls-certificates or public-ca-issued-server-tls-certificates-with-directory-allowlist): Options for authenticating endpoints operated by Members; selection depends on Scheme preference. Both rely on Directory registration of hostnames.
    • TLS posture (baseline-tls-configuration): Baseline cryptographic requirements for all machine-to-machine and public endpoints.

    Authorization and Permission Lifecycle

    • OAuth profile (oauth-with-member-identity-certificates): FAPI 2-based profile binding tokens to Directory URLs via client certificates; uses Registry license URIs as scopes.
    • Permission Records (permission-records): Standard endpoint from OAuth Issuers to expose details of the user-granted permission linked to a token/refresh token, including evidence URLs.
    • Withdrawal of Permission (withdrawal-of-permission): Standard message for notifying data providers of revocations; typically rides over the Message Delivery channel.

    Data Discovery and Publication

    • Data Catalog Records (data-catalog-records): DCAT-based metadata for datasets and data services, referencing Registry schemas, licenses, and scheme requirements.
    • Data Catalog Publishing (data-catalog-publishing): Directory process for ingesting/validating catalog entries from hosted URLs or form uploads. Depends on Data Catalog Records.
    • Open vs. assured data (open-data, assured-open-data): Optional publication profiles for unauthenticated or higher-assurance open datasets.

    Data Exchange, Notifications, and Audit

    • Message Delivery (message-delivery-to-applications): Lightweight best-effort JSON messaging between Applications. Senders discover delivery URLs from the Directory and validate message schemas from the Registry; mTLS uses Member certificates.
    • Provenance Records (provenance-records): Signed, append-only provenance steps transported alongside data; signatures use Directory-issued signing certificates and reference licenses/permission artifacts.
    • Permission-aware access: APIs secured with OAuth and Member certificates check roles (from certificates/Directory), evaluate permission status (Permission Records), and honor revocation notices (Withdrawal of Permission).

    Selecting specifications for a Scheme

    Start by choosing the minimal trust spine: the Registry specs give you authoritative URIs and versioning, the Directory makes them discoverable, and the baseline TLS profile sets a common security floor. Decide how you want to authenticate servers—either via Directory-issued internal CAs or public CAs constrained by an allowlist—then layer Member identity certificates so every client and signer is bound to a Directory URL and roles.

    If your use case involves API calls with fine-grained permissioning, add the OAuth profile plus RBAC. Use machine-readable licenses as scopes so user consent maps directly to a Registry URI, and expose Permission Records so relying parties can check what was granted and when. Include Withdrawal of Permission if you need active revocation signals rather than passive expiry.

    For discoverability, pair the Data Catalog Records format with the Catalog Publishing process. That gives you consistent DCAT-based metadata and a Directory ingestion workflow. Add the open/assured data profiles if you want to advertise open datasets or higher-assurance variants, and use the generic assurance/sensitivity levels if your market needs tiers of trust or handling requirements.

    For coordination and auditability, pick Message Delivery when you need lightweight notifications between applications (data available, permission revoked, workflow events) without overloading core APIs. Choose Provenance Records when lineage and signed hand-offs matter; it reuses the same certificates and Registry identifiers, so provenance chains can be validated without bespoke keys or IDs.

    Scheme-specific customizations live in the Registry: define your roles, your licenses, the schemas for your messages, and the JSON Schema/OpenAPI/DCAT pieces for your data. Once those URIs exist, reference them consistently from Directory entries, OAuth scopes, message subjects, Catalog entries, and provenance steps.

    Working within a Scheme

    When you need to know the meaning of something, start at the Registry: it tells you the authoritative URI and the schema. When you need to know who is allowed to do what, start at the Directory: it lists members, applications, roles, assurance levels, and the endpoints for message delivery. When you need to know what data exists and how to call it, read the Catalog entries published through the Catalog Publishing process.

    Enforcing access follows the same pattern everywhere: verify mTLS with Member certificates, extract the Directory URL and roles, and compare against required roles for an API. In OAuth flows, require that the scope is a Registry license URI; fetch Permission Records to verify what the user granted, and listen for Withdrawal of Permission messages if your Scheme supports active revocation.

    Use Message Delivery for small, time-sensitive nudges—“data ready”, “permission revoked”, “audit event available”—and keep bulk data on your domain APIs, protected by the OAuth profile. Both channels rely on Directory discovery and Registry-defined schemas, so clients can validate payloads and endpoints consistently.

    For audit and lineage, sign provenance steps with Directory-issued signing certificates, embedding the relevant license and permission references. If permissions change, emit Withdrawal of Permission notifications and update downstream provenance as needed, giving relying parties a clear trace of what changed and when.

    Manage change by versioning Registry entries: add new roles or licenses as new URIs instead of mutating old ones. Update Directory and Catalog references to the new URIs while leaving prior versions valid per the Registry versioning rules. This keeps running integrations stable while allowing incremental evolution.

    How the pieces fit in typical flows

    1. Define scheme rules in the Registry (roles, licenses, technical profiles, message/data schemas); publish stable URIs via Registry + versioning.
    2. Enroll participants in the Directory with member/application metadata and roles; issue client/signing certificates and register service hostnames for TLS.
    3. Publish data offerings through Data Catalog Records, ingested via Catalog Publishing; catalog entries reference Registry licenses, schemas, and assurance levels.
    4. Authorize access with OAuth using Member certificates; tokens are bound to Directory URLs and scopes point to Registry license URIs.
    5. Provide evidence via Permission Records and audit trails via Provenance Records; revoke via Withdrawal of Permission messages over Message Delivery.
    6. Notify and integrate using Message Delivery for out-of-band signals (e.g., permission changes, data availability) with mTLS identity derived from Directory certificates.

    This layering lets Schemes compose only the specifications they need while maintaining consistent identifiers, discovery, identity, and auditability across the Trust Framework.
