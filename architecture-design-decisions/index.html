    Design Decisions Behind the IB1 Specifications

    This note captures the architectural choices and reasoning that shaped the IB1 specifications: the problems we needed to solve, the risks we wanted to avoid, and why particular options won. Schemes remain free to pick only what they need; dependencies are explicit.

    Unifying principles

    • Stable URIs as anchors: We needed identifiers that survive version bumps, signatures, and cross-organisation use. Registry URIs keep agreements explicit, allow caching, and avoid per-implementation enums. We rejected opaque, issuer-local IDs because they break validation across parties and make signed artifacts fragile when identifiers change.
    • Directory as the single discovery hub: Fragmented onboarding and bilateral spreadsheets do not scale. Centralising “who/where/what role/what assurance” in the Directory reduces integration friction and keeps assurance decisions consistent. Per-provider portals were rejected because they multiply integration cost and make role/assurance checks incomparable.
    • Certificates carry identity and role: By embedding the Application URL and roles in the certificate, every mTLS session already contains who the caller is and under which roles, eliminating a second client registry and easing rotation (rotate cert, identity unchanged). Shared secrets and static client IDs were rejected for weaker security, key sprawl, and poor non-repudiation.
    • Transport-agnostic semantics: We wanted one set of meanings for licenses, permission facts, provenance, and messages, regardless of whether the channel is REST, push, or file. Protocol-specific schemas were rejected because they diverge quickly, force duplicate audit logic, and complicate assurance reviews.
    • Modular, minimal dependencies: Schemes vary in market/regulatory needs. Breaking capabilities into small specs with explicit dependencies lets each Scheme pick the smallest sufficient set without rebuilding the trust spine. A monolithic profile was rejected to keep adoption tractable and avoid over-prescription where markets differ.

    Identity, security, and transport choices

    • TLS 1.3 baseline (baseline-tls-configuration): Picked for forward secrecy, simpler cipher suite surface, and consistent posture across participants. Older versions were excluded to avoid downgrade risk and configuration variance that complicates audits.
    • Two server-certificate options (directory-issued-server-tls-certificates vs public-ca-issued-server-tls-certificates-with-directory-allowlist): Some deployments can install a private root and want strong Directory-bound issuance; others need public CA trust chains. Offering both preserves interoperability while keeping the same discovery/identity semantics. A single option was rejected to avoid blocking participants with differing infra constraints.
    • Member certificates for clients and signing (member-identity-digital-certificates): Chosen to give a durable, role-bearing identity that works for both online calls and offline signatures. JWT client assertions were deferred because certs already solve key distribution for mTLS and signing with one mechanism, reducing complexity for operators.

    Authorization and consent decisions

    • OAuth profile bound to Directory URLs (oauth-with-member-identity-certificates): We needed mainstream interoperability (hence FAPI 2) but with strong binding to the Directory identity. tls_client_auth plus PKCE reduces credential theft risk; using license URIs as scopes ties consent text to a stable legal reference. Dynamic client registration and secrets were dropped because the certificate already proves client identity; mandatory JWT tokens were dropped to avoid unnecessary coupling to token format.
    • Permission Records (permission-records): Tokens alone do not tell you the full permission fact pattern or evidence. A separate endpoint lets relying parties fetch authoritative, up-to-date details and keep tokens small. Packing it into JWTs was rejected due to size, stale data risk, and awkward evidence URL handling.
    • Withdrawal of Permission: Chosen to give an active revocation path without forcing tight token lifetimes. Short-lived tokens alone were rejected because many data exchanges need long-lived refresh tokens but still require explicit revocation signals.

    Data discovery and publication

    • DCAT-based Catalog (data-catalog-records, data-catalog-publishing): We wanted existing tooling, clear linkage to licenses/schemas, and multi-format RDF support. DCAT provided all three. Publishing through the Directory keeps provenance and auditing simple. A bespoke JSON format was rejected to avoid fragmenting the ecosystem and to leverage DCAT validators and SPARQL.
    • Open and assured profiles (open-data, assured-open-data): Added to let Schemes surface open datasets and signal assurance without changing core semantics. This balances openness with trust signaling; omitting profiles was rejected because markets differ on openness requirements.
    • Assurance/sensitivity vocabularies (generic-organizational-assurance-levels, generic-dataset-assurance-levels, generic-sensitivity-classes): Optional, because some markets demand graded handling rules. Not baking them into the core keeps Schemes lean if they do not need tiers.

    Messaging, notifications, and audit

    • Message Delivery (message-delivery-to-applications): We needed lightweight, low-friction signals (availability, revocation, workflow events) without inventing new APIs per pair. Best-effort and idempotence keep implementations simple; retries cover transient failure, and Schemes can set SLA expectations if needed. Reliable queues were avoided to keep cost/complexity low for Schemes that only need hints.
    • Provenance Records (provenance-records): Chosen to give verifiable, append-only lineage across organisations without a central store. By reusing cert identity and Registry URIs, signatures remain meaningful over time. A central audit log was rejected to avoid single points of failure, privacy concerns, and cross-border data residency issues.

    Evolution and change control

    • Registry versioning (registry-versioning, registry-process-resources): We needed to evolve roles, licenses, and schemas without breaking older references or signed artifacts. Issuing new URIs rather than mutating old ones keeps history intact and lets integrations upgrade at their pace.
    • Cache-friendly discovery: Encouraging caching for Directory/Registry responses and stable delivery URLs during cache lifetimes reduces load and flakiness. Forcing clients to re-fetch on every call was rejected to avoid latency, fragility, and avoidable outages during transient Registry/Directory issues.

    When to include or skip components

    • Message Delivery: Include if you need near-real-time cross-participant signals without tight coupling to provider APIs. Skip if you already have reliable eventing and do not need framework-level messaging.
    • Permission Records / Withdrawal of Permission: Include for end-user mediated data or explicit revocation needs. Skip if flows are purely B2B under fixed contracts with no per-user consent or revocation semantics.
    • Provenance Records: Include when verifiable lineage and signed hand-offs matter (compliance, multi-hop processing). Skip if data never leaves one operator or audit demands are low.
    • Assurance/sensitivity vocabularies: Include if the market benefits from standardized handling tiers; skip to keep minimal surface if roles/licenses are sufficient.
